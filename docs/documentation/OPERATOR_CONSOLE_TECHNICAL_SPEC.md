# PROMETHEUS Operator Console Technical Specification

## Release identity

- Product: PROMETHEUS V-1.1.1 Competition Edition
- Surface: Operator Console
- Mission: PROM-COMP-001
- Mode: deterministic visualization of a verified public snapshot
- Canonical sequence: PROMETHEUS → AIS-Ω → Adversarial Twin → SECA → HYDRA → ProofGrid → Capability Genome → Build Truth

## Product objective

The Operator Console presents one governed software mission as a synchronized technical and theatrical experience. The page must show what happened, which agent acted, what packet moved, what code behavior changed, what evidence accumulated, what claim remained blocked, and why Build Truth ultimately authorized promotion.

The console is not the local execution runtime. It is the public observability and verification surface for evidence generated by the executable competition package.

## State contract

Every major component reads from one mission state:

- mission and run identifiers
- current stage and active agent
- packet type and packet identifier
- handoff source, destination and acknowledgement
- proof confidence and orchestration load
- latency, packet count, queue depth and handoff count
- tests, artifacts, claims, blockers and receipt state
- Capability Genome state
- release state

A stage change updates terminal events, code panes, handoff nodes, packet position, global telemetry, agent visualizations, evidence, theater state and final verdict together.

## Operator controls

- Run: execute the complete eight-stage replay
- Pause and resume: hold or continue the current run
- Step: advance exactly one governed stage
- Reset: return every surface to the staged state
- Speed: 0.5×, 1×, 1.5× or 2× replay timing
- Verified state: load the complete evidence-supported final state
- Theater: open the full-screen judge presentation

## Global telemetry

The global chart contains four deterministic series:

1. Proof confidence
2. Evidence completeness
3. Claim support
4. Release readiness

Release readiness intentionally falls to zero at the SECA denial stage. It rises only after HYDRA changes behavior and ProofGrid verifies the resulting evidence.

Additional telemetry includes orchestration load, handoff latency, packet count, handoff count, queue depth, test totals, artifact totals, claim support, blocker count, receipt state, Genome state and release state.

## System visualization specifications

### PROMETHEUS

Presentation: mission orchestration trace.

Purpose: show mission completeness, acceptance-gate coverage, packet dispatch and campaign phase. PROMETHEUS remains the coordinating authority across the entire replay and does not reach completion until Build Truth authorizes promotion.

### AIS-Ω

Presentation: three-branch route evaluation.

Purpose: make candidate generation visible. route-a and route-b are rejected while route-c remains selected. The visualization emphasizes distinct alternatives rather than a single opaque answer.

### Adversarial Twin

Presentation: contradiction-pressure trace.

Purpose: show probes, failure reproducibility, defect severity and contradiction pressure. The graph spikes when the duplicate consequential operation is reproduced and remains part of the evidence record after repair.

### SECA

Presentation: locked promotion gate.

Purpose: display fail-closed governance. The gate remains closed after failure and through repair. It is not reopened by a patch alone; ProofGrid verification is required.

### HYDRA

Presentation: before-and-after behavioral delta.

Purpose: contrast `ReplayDetected` with `replay_safely_ignored`. The console also displays the code transformation that introduces deterministic payload comparison and idempotent replay handling.

### ProofGrid

Presentation: stepwise evidence accumulation.

Purpose: display 5/5 tests, 8/8 artifacts, complete hash verification and a sealed receipt. Progress advances in discrete proof events rather than decorative motion.

### Capability Genome

Presentation: applicability and inheritance map.

Purpose: display the trigger, applicability, exclusion and related-task reuse of PG-CG-REPLAY-GUARD-001. The result is measured as one failed action reduced to zero.

### Build Truth

Presentation: canonical release ledger.

Purpose: show supported claims, contradictions, blockers and release readiness. Build Truth is the final authority and does not authorize release until the complete proof chain closes.

## Terminal contract

Each line includes elapsed time, event type, source and message. Handoff events additionally include source, destination, packet type, packet ID and acknowledgement.

Primary event types:

- READY
- GUARD
- ADMITTED
- ROUTED
- FAILED
- BLOCKED
- REPAIRED
- VERIFIED
- PRESERVED
- AUTHORIZED
- HANDOFF
- SEALED
- PROMOTION

## Code presentation

The code comparison shows the unsafe replay path and the verified idempotent path. It exposes the behavior change required to understand the demonstration while excluding broader proprietary implementation details.

## ServerForge and EDEN integration

EDEN is described as the local execution body: the Windows-based project workspace, runtime and repository environment where Codex and PROMETHEUS performed the engineering work.

ServerForge is the observability transport. It records sanitized campaign events, recoveries, publications and evidence lineage for the public case study. It is not added to the canonical eight-agent proof chain.

## Intellectual-property boundary

The public release may disclose:

- system roles and state transitions
- high-level execution topology
- test outcomes and artifact counts
- proof receipts and hashes
- bounded code behavior needed to understand the demonstration
- sanitized case-study events

The public release must not disclose:

- private prompts or orchestration instructions
- proprietary routing or recursive self-modification mechanisms
- credentials, webhooks or tokens
- private machine paths or local state
- private repositories
- patent-sensitive algorithms
- unrelated estate architecture

## Failure behavior

- Data load failure: no completion claim is displayed.
- Missing receipt: release remains locked.
- Hash mismatch: ProofGrid integrity failure and promotion revocation.
- Reduced motion: transitions become discrete while preserving all information.
- Static hosting: all public asset paths remain relative and project-Page safe.

## Acceptance gates

- exactly eight canonical agents in exact order
- no visible naming drift
- no random telemetry
- all operator controls present
- terminal, code delta, handoff fabric and packet runway synchronized
- four global telemetry series present
- eight role-specific system visualizations present
- EDEN and ServerForge case study present
- explicit IP boundary present
- full-screen theater and verified-result path functional
- five tests and eight artifacts represented accurately
- receipt identity and hash aligned with the public ProofGrid receipt
- no broken local references or duplicate HTML identifiers
